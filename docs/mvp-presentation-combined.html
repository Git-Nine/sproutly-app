⚙ The method

How it was built

I direct AI assistants through a fixed sequence of steps, approving the work at every stage. Same recipe for every feature. Nothing ships unchecked.

02 / 08 · Two threads

The workflow

Every feature travels the same seven gates.

01/initDecide what the product is — the goals and the feature list.
02/write-specWrite down exactly what one feature should do, before building.
03/architectureDecide how it'll work technically — still no code.
04/frontendBuild the screens people actually see and tap.
05/backendBuild the engine, the data, and the security behind it.
06/qaTest that it works — and that it's safe.
07/deployPut it live for real people to use.

I sign off at every gate before the next one begins.

03 / 08 · The pipeline

What each stage really does

Decide, build, prove, ship.

Stage · Decide

Agree what to build, and how

Pin down the idea and the plan on paper before a single line of code exists.

/init · /write-spec · /architecture — a product brief, one testable spec per feature, then the technical design.

Stage · Build

Make the thing

Turn the agreed plan into working screens and the machinery behind them.

/frontend · /backend — UI from a shared component library, then APIs, database schema and access rules.

Stage · Prove

Check it works and is safe

Don't take "it's done" on faith — confirm it against what was agreed.

/qa — tested against the spec's acceptance criteria, plus a dedicated security audit.

Stage · Ship

Put it in people's hands

Release it to the real world with the final safety checks done.

/deploy — pushed to production on Vercel with production-readiness checks.
04 / 08 · The stages

Why it doesn't go off the rails

Speed, without losing control.

I approve every stage

The AI never jumps ahead. I review and say "go" before each step.

No phase advances without explicit sign-off.

One feature at a time

Small, focused pieces of work — easy to check, hard to get lost in.

Single-responsibility specs keep every change reviewable.

The spec is the agreement

I write down what "done" means first, so there's no arguing later.

Acceptance criteria written before code → testing is objective.

The assistant proves its work

It shows that changes actually landed, instead of just claiming success.

Write-then-verify: re-reads files after editing — no hallucinated "done".

06 / 08 · The result

Built to grow

Smart today. Smarter later — without a rebuild.

Right now Sproutly makes its choices with clear, dependable rules. The workflow built it so that, later, AI can take over those same choices — without tearing anything down. Same screens, same data, smarter brain.

07 / 08 · Future-proof

The takeaway

One disciplined workflow, two payoffs.

🌿 For everyone

A real, working product — built quickly, checked carefully, and live today.

⚙ For builders

  1. Specs before code
  2. Small, single-responsibility units
  3. Gate every phase with a human
  4. Give the AI persistent memory
  5. Bake in write-then-verify
  6. Design seams for AI swap-in

Direct the work. Approve at the gates. Ship something real.
