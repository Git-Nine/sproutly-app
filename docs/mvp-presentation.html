02 / 09 · The product

The workflow · one feature through seven gates

Each feature runs the same pipeline. No vibe coding.

01/initDefine the product — PRD + prioritised feature map. Run once.
02/write-specOne feature, one full spec. Acceptance criteria written before code.
03/architectureDesign the tech approach. PM-friendly, no code yet.
04/frontendBuild the UI. Installed shadcn/ui components first.
05/backendAPIs, database schema, row-level security rules.
06/qaTest against acceptance criteria + run a security audit.
07/deployShip to Vercel with production checks.

/refine PROJ-X — revisit and challenge any spec, at any point.

03 / 09 · The pipeline

What actually keeps an agent on task

Five things that turn "generate code" into reliable delivery.

Specs are the contract

Acceptance criteria exist before any code, so QA tests against something objective — not vibes.

One feature per spec

Small, bounded units keep the agent focused and the output reviewable in one sitting.

Humans gate every phase

The agent never jumps a phase without approval. You direct; it executes.

Specialised sub-agents

Frontend, Backend and QA are separate agents with their own tools and scope — right context for the job.

Persistent memory

features/INDEX.md is a status state-machine the AI reads at start and writes when done — so it never loses the thread across sessions.

The result

You stay the architect. The agents do the typing. Nothing ships that a human didn't approve at a gate.

04 / 09 · Directing

Guardrails · steal these for your own projects

Conventions that keep agents on the rails.

ConventionWhy it helps when directing AI
PROJ-1, PROJ-2…Stable IDs both human and agent reference without ambiguity.
feat(PROJ-X):Every commit traces straight back to a spec.
write-then-verifyAgent re-reads a file after editing to confirm the change landed — kills "I updated it" hallucinations.
read before editNever modify from memory; re-read after any context compaction.
shadcn/ui firstNever re-invent installed components — less surface area to get wrong.
tests co-locatedUnit tests next to source, E2E in tests/. QA is built in, not bolted on.
05 / 09 · Guardrails

Under the hood

The stack, and a security model that's enforced — not aspirational.

Next.js 16 · App Router TypeScript Tailwind + shadcn/ui Supabase · Postgres/Auth/Storage Zod + react-hook-form Vercel
  • Row Level Security on every user table — user_id = auth.uid(). You can only ever touch your own rows.
  • Private, user-namespaced storage for photos — /{user_id}/file.
  • Role-gated admin routes — role = 'admin' for the plant catalogue.
A real QA catch: a Zod .url() validator was accepting javascript: and data: URLs. Caught in /qa, restricted to http(s). That's the security-audit phase earning its keep.

07 / 09 · Shipped

The killer detail · designed for AI swap-in

Rules today. Models tomorrow. Same database, same frontend.

Stagev1 — built nowSwap-in — later, same output
ScanEXIF + manual formVision model fills the same fields

The lesson for AI engineers: separate concerns so the AI layer is a swap, not a rewrite. Define the interface first — the model is just one implementation behind it.
